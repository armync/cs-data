Introducere

Internetul din ziua de azi este compus din peste 45,000 de reele diferite numite Autonomous Systems (AS). Un AS este reprezentat de un set de reele ce aparin aceleiai organizaii sau ri. Exemple mari de AS-uri sunt Google, Facebook, Sprint. Chiar i RoEduNet este un AS. Aici gsii o hart interactiv cu AS-urile prezente n momentul de fa in internet, iar aici gsii detalii despre AS2614 (AARNIEC / RoEduNet).

AS-urile pot fi mprite n dou clase mari, AS-uri de tranzit i stub. Un AS stub primete/trimite pachete n care sursa sau destinaia sunt IP-uri din acest domeniu (e.g. RDS va primi/trimite pachete n care sursa sau destinaia n header-ul IP este un IP pe care RDS l deine). n schimb, un AS de tranzit este un AS care ofer servicii de rutare altor domenii. Mai exact, ntr-un AS de tranzit sunt pachete a cror surs i destinaie nu aparin neaprat AS-ului. Astzi, peste 85% dintre AS-uri sunt de tip stub. n imaginea de mai jos 22, 87264, 57 i 4561 sunt stub, iar restul sunt tranzit. Mai mult de att, AS-urile 57 i 87264 sunt multihomed.

Internet eXchange Points (IXP)

O soluie folosit astzi pentru a permite unui numr mare de AS-uri s se interconecteze este crearea de Internet eXchange Points (IXP). n general, un IXP reprezint o grupare de routere, aparinnd unor AS-uri diferite, colocate i interconectate ntr-o locaie neutr, precum un datacenter. Practic, pentru fiecare AS care vrea s participe la interconectare, va exista un router n acest datacenter care e legat la AS-ul din care acesta face parte i la un switch din datacenter (vor fi n acelai LAN i vor putea trimite informaii ntre ele). La acesta sunt legate i alte routere din domenii care vor s se interconecteze.

Factorul economic

n primele zile ale internetului, domenilor schimbau ntre ele toate rutele tiute. Totui, internetul din ziua de azi are la baza factorul economic. ntre AS-uri exist mai multe tipuri de relaii economice:

  • customer - provider
  • shared-cost peering

n imaginea de mai sus, AS7 pltete AS4 pentru a l folosi ca un AS de tranzit. Legturile cu $ sunt de tip customer provider. n schimb legturile mov sunt de tip shared-cost peering. De exemplu, AS3 i AS4 ambele pltesc AS1 pentru a l folosi ca AS de tranzit. Astfel, entitile care dein AS3 i AS4 pot alege s fac peering ntre ele i s plteasc doar un link cu AS1.

Protocolul BGP

n ziua de astzi este folosit un singur protocol pentru rutare ntre AS-uri, Border Gateway Protocol (BGP). BGP ruleaz peste TCP i este un protocol de tip path vector (vectorul distanelor, similar cu RIP). Fiecare AS este reprezentat de un ID (integer) unic. n cazul anunurilor de rute, BGP anun un prefix, ct i drumul pn la acesta. Acest drum reprezint o list de AS-uri. Astfel este evitat problema numrrii la infinit.

n mod istoric, BGP este un protocol care se baza pe ncredere. Acest lucru a fcut posibile incidente precum:

  • Un ISP din Pakistan care a anunat, n 2008, rute pentru Youtube n internet, tind accesul la acesta. https://www.wired.com/2008/02/pakistans-accid/
  • Un ISP din Rusia a rutat traficul serviciilor bancare (Visa, Mastercard) prin Rusia. https://arstechnica.com/information-technology/2017/04/russian-controlled-telecom-hijacks-financial-services-internet-traffic/
  • AS7007 incident

n prezent, se ncearc adoptarea RPKI pentru validarea folosind certificate a update-urilor BGP.

Tipuri de mesaje
  • OPEN: mesajul de OPEN este trimis imediat ce conexiunea TCP dintre dou routere este realizat. Acest mesaj iniializeaz sesiunea BGP i faciliteaz negocierea mai multor opiuni pentru protocol.
  • NOTIFICATION: mesajul NOTIFICATION este folosit pentru a nchide sesiuni BGP. n general, acest mesaj este trimis atunci cnd apar erori.
  • UPDATE: mesajul este folosit pentru a anuna noi rute sau modificri la rutele existente. Spre deosebire de ali algoritmi de tip vector distance, BGP trimite doar update-uri cu schimbri, nu ntreaga tabel de rutare la fiecare update.
  • KEEPALIVE: mesajul este trimis regulat i este folosit pentru a confirma c routerul peer este conectat. n cazul n care pentru o perioad de 90 de secunde nu primim un mesaj de tip KEEPALIVE de la un peer, nseamn c sesiunea este down i trebuie s anunm retragerea tuturor rutelor prin acest peer.

Gsii un exemplu de captur de mesaje BGP n urmtoarea captur de pe Cloudshark.

Extensii BGP

n implementarea protocolului, pentru a fi mai uor de configurat, avem dou mari extensii:

  • External Border Gateway Protocol (eBGP) - Extensie a protocolului BGP folosit pentru comunicarea n exteriorul unui AS.
  • Interior Border Gateway Protocol (iBGP) - Extensie a protocolului BGP folosit pentru comunicarea n interiorul unui AS. Practic, rutele care sunt aflate folosind EBGP vor fi sumarizate n AS folosind iBGP.

Topologie

n topologia laboratorului curent, fiecare grup va administra un AS cu un spaiu de adrese de forma X.0.0.0/8, unde X este numrul grupei. Acest spaiu de adrese este mprit conform topologiei de mai jos, astfel:

  • Legtura cu IXP-ul va avea un IP din reeaua 180.22.0.0/24, de forma 180.22.0.X, unde X este numrul grupei / AS-ului. Spre exemplu, AS 13 va avea pe interfaa ctre IXP IP-ul 180.22.0.13. IXP-ul va avea IP-ul 180.22.0.22.
  • Subreeaua dintre LOND i PARI va folosi spaiul X.0.1.0/24
    • X.0.1.1 pentru LOND
    • X.0.1.2 pentru PARI
  • Subreeaua dintre LOND i BERL va folosi spaiul X.0.2.0/24
    • X.0.2.2 pentru LOND
    • X.0.2.1 pentru PARI
  • Subreeaua dintre PARI i BUCH va folosi spaiul X.0.3.0/24
    • X.0.3.1 pentru PARI
    • X.0.3.2 pentru BUCH
  • Subreeaua dintre BERL i BUCH va folosi spaiul X.0.4.0/24
    • X.0.4.2 pentru BERL
    • X.0.4.1 pentru BUCH
  • Subreeaua dintre BUCH i host va folosi spaiul de adrese X.104.0.0/24

Mai mult, fiecare router va avea configurate urmtoarele adrese de loopback:

  • X.151.0.1 pentru LOND
  • X.152.0.1 pentru PARI
  • X.153.0.1 pentru BERL
  • X.154.0.1 pentru BUCH

01. [10p] Conectare la infrastructur

n cadrul laboratorului vei lucra n echipe. Fiecare echipa va administra propria reea virtual. La nceputul laboratorului vei primi un numr de grupa i parol de ssh pentru acces la reeaua voastr. Pool-ul de adrese asociat reelei va fi sub form <numr grupa>.0.0.0/8. Fiecare echip are acces la reea printr-un container bastion. Pentru a accesa acest container putei folosi comanda: 

ssh -J your_user.here@fep.grid.pub.ro -p 2000+<numr grup> root@10.9.2.74

De exemplu, studentul John Smith din grupa 11 se va conecta folosind: 

ssh -J john.smith@fep.grid.pub.ro -p 2011 root@10.9.2.74

O dat conectai v vei afla pe container-ul bastion (ssh), de aici putei apela scriptul ./goto.sh pentru a v conecta la routere i host-uri. De exemplu, pentru a v conecta la routerul BUCH vei apela: 

./goto.sh BUCH router

Aceast comand v va oferi acces la terminalul router-ului, cu o interfa similar cu cea CISCO IOS.

Reminder comenzi generice

  • configure terminal - intr n modul de configurare; orice comand dat n modul de configurare va fi aplicat asupra router-ului.
  • ping <destination> - trimite pachete de tip ICMP ctre <destination>.
  • traceroute <destination> - afieaz calea pe care o ia un pachet pentru a ajunge la <destination>.
  • exit - iese din modul curent de configurare.

Pentru a accesa host-ul conectat la router-ul BUCH vei apela: 

./goto.sh BUCH host

02. [20p] Configurare eBGP

Vom configura eBGP pe router-ul LOND. Vom exporta ctre IXP reeaua noastr (X.0.0.0/8) i vom importa tot ce primim de la IXP. IXP-ul este configurat astfel nct s exporte toate rutele primite de la AS-urile vecine. Pentru conectarea la IXP vom folosi reeaua 180.22.0.0/24.

Pentru a activa i configura procesul de BGP al router-ului vom folosi comanda de mai jos: 

LOND_router# configure terminal
LOND_router(config)# router bgp <X>
LOND_router(config-router)#

unde <X> este numrul grupei/AS-ului. Observm schimbarea prompt-ului, sugernd modul de configurare actual.

n general, avem opiunea s configurm un router-id. Acest lucru poate influena criteriile de alegere a rutei n BGP. Pentru moment, nu e neaprat necesar, dar l vom seta ca fiind IP-ul ctre IXP. Pe interfaa ixp_22 (ctre IXP) a router-ului LOND este deja configurat o adres IP. Pentru a afla adresele IP de pe interfee putem folosi comanda de mai jos:

03. [20p] Configurare iBGP

n cadrul acestui exerciiu vom configura iBGP pe router-ele LOND i PARI.

Comenzile sunt similare, ns, de aceast dat, nu e nevoie s configurm un router-id. De asemenea, vom folosi adresele interfeelor de loopback (lo) pentru a face peering cu router-ele din acelai AS.

Dac v ntrebai cum putem avea conectivitate folosind adresele de loopback, acest lucru este posibil datorit faptului c pe routere ruleaz i protocolul OSPF, care este configurat s propage rute ctre aceste adrese de loopback. 

PARI_router# show ip ospf route
============ OSPF network routing table ============
<...>
N    21.151.0.1/32         [1] area: 0.0.0.0
                           via 21.0.1.1, port_LOND
N    21.152.0.1/32         [0] area: 0.0.0.0
                           directly attached to lo
N    21.153.0.1/32         [2] area: 0.0.0.0
                           via 21.0.1.1, port_LOND
                           via 21.0.3.2, port_BUCH
N    21.154.0.1/32         [1] area: 0.0.0.0
                           via 21.0.3.2, port_BUCH
<...>

n general, o sesiune de BGP e stabilit pe interfaa cea mai apropiat de vecin. Pentru ca rutele s fie acceptate, va trebui s schimbm interfaa surs ctre cea de loopback.

Spre exemplu pentru a face peering ntre router-ele LOND i PARI vom da urmtoarele comenzi: 

LOND_router# configure terminal
LOND_router(config)# router bgp <X>
LOND_router(config-router)# neighbor <X>.152.0.1 remote-as <X>
LOND_router(config-router)# neighbor <X>.152.0.1 update-source lo
LOND_router(config-router)#
PARI_router# configure terminal
PARI_router(config)# router bgp <X>
PARI_router(config-router)# neighbor <X>.151.0.1 remote-as <X>
PARI_router(config-router)# neighbor <X>.151.0.1 update-source lo
PARI_router(config-router)#

Mai multe informaii despre acest mod de configurare gsii aici.

Verificai dac s-a realizat conexiunea BGP, folosind comanda show bgp summary.
