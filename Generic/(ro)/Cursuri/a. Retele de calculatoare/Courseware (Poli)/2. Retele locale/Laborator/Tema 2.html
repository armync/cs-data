Tema const n realizarea configuraiei unui set de exerciii pe o topologie (vedei mai jos) simulat folosind containere (implementare n ContainerNet + Docker) ntr-o main virtual (ori n cloud - OpenStack, ori descrcat i rulat local).

Notare

Fiecare exerciiu are un punctaj propriu. Nota pe ntreaga tem este dat de suma punctajelor acumulate n urma rezolvrii fiecrui exerciiu.

Punctajul maxim care se poate obine pe ntreaga tem este 100 de puncte. Acest punctaj este echivalent cu 2 puncte din nota final.

Exist i exerciii bonus, cu ajutorul crora putei obine 125 de puncte.

Nu este obligatorie rezolvarea tuturor exerciiilor. Exerciiile pot fi rezolvate n orice ordine, mai puin n situaia n care un exerciiu depinde de rezolvarea unui alt exerciiu (de obicei, primele 4 de stabilire a conectivitii containere Internet).

Predarea temei

  1. Pentru mpachetarea soluiei, rulai: 
    root@host$ t2check --save

    (atenie: aceast comand va reseta reelistica i reporni toate serviciile, echivalent cu un reboot mai rapid asigurai-v c ai lucrat persistent (vedei mai jos, la task-uri, cum)!)

  2. Rezultatul este o arhiv semnat n directorul n care invocai comanda ($PWD).
  3. Folosind utilitarul scp, copiai acest fiier pe staia voastr local (atenie s nu ncurcai directoarele i s copiai o arhiv veche / salvat n alt parte!). Dac suntei conectat la OpenStack prin serverul intermediar fep.grid.pub.ro, va trebui s copiai mai nti acolo, apoi s-o preluai pe staia voastr de lucru (alternativ, putei folosi funcionalitatea de ProxyCommand a clientului ssh pentru o conexiune direct).
  4. ncrcai arhiva pe RL Checker (Moodle).

Este obligatoriu ca rezolvarea exerciiilor s se fac n mod persistent. La o repornire a mainii virtuale, rezolvrile trebuie s rmn active, altfel putei ntmpina dificulti la o revenire ulterioar asupra temei.

Folosii comanda reboot nainte s testai de-a ntregul (checkerul local doar simuleaz ceva mai rapid).

De asemenea, putei folosi comanda systemctl restart rl-topology pentru a reporni rapid toate containerele (sistemul lor de fiiere este i el persistent).

Pentru a rezolva tema este suficient s prelucrai fiiere doar din cile /etc, /home i /root (att pe host, ct i pe containere). NU instalai alte pachete n plus!

NU oprii / tergei manual containerele Docker (dect dac dorii s luai de la zero cu configurarea acestora, vedei mai jos comanda).

Pentru a terge i reseta configuraia de pe un container, se folosii secvena: 

systemctl stop rl-topology && docker container rm <nume container> && systemctl restart rl-topology;

Topologie & infrastructur

n topologie avei ruterele host (fiind VM-ul care gzduiete toat infrastructura), Router0 (containerul are denumirea R0) i Router-X (sub ID-ul R-X), PC1, PC2, PC3 conectate la SW (switch nu vei avea nimic de configurat la el) prin 2 VLAN-uri (10 i 20) i PC-X (reea cu R-X).

Atenie: din motive tehnice, identificatorii echipamentelor Router0 i Router-X au fost prescurtate la R0 respectiv R-X. Folosii denumirile prescurtate DOAR ATUNCI cnd v conectai la container (folosind comanda go sau comenzi docker dac va fi cazul), n rest utilizai denumirea full pentru orice altceva (e.g., alias / hosts).

Fapt divers: VLAN9 se numete reeaua de pe OpenStack, dac rulai local, acolo va fi reeaua hivervizorului (VMWare / VirtualBox) cu maina fizic.

Va trebui s realizai primele 4 exerciii n ordine. ntruct aceste exerciii ofer, n final, conectivitate la Internet, restul vor depinde de acestea!

Configuraii persistente

Toate configuraiile s fie persistente. Trebuie s fie active i dup repornirea mainii virtuale.

Pentru modificarea unor fiiere speciale din containerele Docker (/etc/hosts, /etc/resolv.conf) va trebui s folosii un script de init sau hook de ifupdown-ng. Gsii meniuni adecvate la exerciiile unde va fi necesar s facei asta!

Pentru configuraie persistent, att host-ul, ct i containerele vin cu ifupdown-ng pre-instalat.

Pentru a ncepe s configurai, verificai fiierele existente la cile /etc/network/interfaces i /etc/network/interfaces.d/* ( RTFM here ;)).

Pentru a configura persistent rute adiionale (i nu numai!), putei folosi hook-ul up din sintaxa de configurare interfaces. Exemplu (fragment): 

iface <intf>
    address A.B.C.D/xx
    up ip route add X.Y.Z.T/yy via Q.W.E.R

Fiierele /etc/network/interfaces* se parseaz i execut linie cu linie, scripturile if[up|down] oprindu-se la prima eroare ntlnit. De exemplu, dac avei ntr-o seciune iface un hook up ip ro add invalid route i pe rndurile urmtoare avei alte declaraii, acestea nu vor mai ajunge s fie aplicate!

Putei folosi urmtorul oneliner pentru a verifica rapid o interfa: 

ifdown --force <intf>; ifup <intf>; ip a sh; ip ro sh
# hint: folosii ifdown i ifup cu parametrul -a pentru a porni TOATE interfeele declarate!
# NS: NU RULAI `ifdown -a` pe host! V VEI PIERDE CONECTIVITATEA PE eth0 !!!

Pentru a salva regulile iptables, urmai paii de aici: https://www.serveracademy.com/courses/linux-fundamentals/how-to-save-iptables-rules-permanently/ (exist mai multe modaliti, e.g. putei pune hook-uri de up la o interfa etc.). SUB NICI O FORM S NU INSTALAI PACHETELE DESCRISE DIN TUTORIALE (mai ales ifupdown v stric VM-ul, avei deja ifupdown-ng!!).

Dac folosii mai multe fiiere n scripturi (e.g., apelai dintr-un script alt script), folosii ci absolute. Adic folosii /root/scripts/make-juju.py n loc de ./make-juju.py pentru a nu se baza pe directorul actual de lucru (working directory). NU uitai s le facei executabile i s includei shebang-ul!

Ex. 1 [25p] Adresare + rutare IPv4

  1. Configurai cu adrese IPv4 toate legturile din topologie, astfel:
    • Subnetai n dou reele fixe (de dimensiuni egale) spaiul 10.13.$A.0/24; distribuii prima subreea ctre VLAN10, iar cealalt lui VLAN20; atribuii prima adres asignabil ruterului (Router0), apoi urmtoarele n ordinea denumirilor staiilor (i.e., Router0, PC1, PC3 respectiv Router0, PC2)! Atenie: pe router avei deja declarate VLAN-urile n fiierul din /etc/network/interfaces.d/!
    • Pentru reelele host Router0 i host Router-X, subnetai optim spaiul 10.11.$B.0/29 i distribuii-le n ordinea meniunii din enun; host trebuie s aib, mereu, prima adres asignabil!
    • Pentru reeaua Router-X PC-X: 172.18.$C.64/29; ruterul va avea prima adres asignabil, iar staia PC-X PE ULTIMA (atenie: nu broadcast!, cea dinaintea acestuia)!
    • Recomandare: nu modificai nimic pe containerul SW (switch), altfel riscai s v stricai infrastructura!
  2. Configurai rutarea IPv4 (att default gateways, ct i rute statice!) astfel nct toate staiile s se poat accesa unele pe altele prin adres IP!

Atenie mare: adresa IP de pe eth0 a sistemului host este asignat dinamic de ctre hipervizor, prin DHCP; NU v atingei (never go full ifdown!) de aceast interfa, altfel riscai s v pierdei accesul la VM!

Ex. 2 [15p] Adresare IPv6

  1. Configurai adrese IPv6 pentru reeaua VLAN10 i VLAN20 (not: variabila $VLANID va avea valoarea 10, respectiv 20):
    • Folosii spaiul 2023:E666:$B:$A:$VLANID::/80.
    • Aceeai ordine de mai sus (ca la IPv4): Router0 va avea prima adres asignabil, apoi PCn sortate numeric.
  2. Configurai conectivitate IPv6 ntre Router0 i host:
    • Folosii spaiul FEC0:5017:$C:$D::/64.
    • Prima adres asignabil este pentru host, a doua a lui Router0.
  3. Configurai rutarea IPv6 pentru a permite comunicarea ntre toate sistemele cu adres IPv6.
    • Reeaua Router-X PC-X NU va avea adres IPv6 ;)

Ex. 3 [5p] Hosts

  • Realizai configuraiile necesare astfel nct echipamentele s poat fi accesate prin numele lor (folosii numele host, Router0, PC1, PC2, PC3, Router-X, PC-X - atenie la MAJUSCULE!). Adugai intrri doar pentru adresele IPv4.

Fiierul /etc/hosts din containere este mai special (montat ca volum --bind de ctre Docker). Acest lucru face ca orice modificare a acestuia s se piard la fiecare restart al containerului (aka reboot al VM-ului sau restart al serviciului rl-topology). Dac (i sigur) vrei s persiste, putei s-l salvai n alt cale (e.g., /etc/hosts.orig i s-l restaurai mereu cnd pornete containerul printr-un hook de up n /etc/network/interfaces). Ah, i nu folosii cp, folosii cat + redirectare bash (fiind volum, nu putei terge + recrea fiierul, trebuie trunchiat & suprascris), e.g.: 

iface <intf>
	up cat /etc/hosts.orig >/etc/hosts

Ex. 4 [5p] Internet connectivity

  • Realizai configuraiile necesare pentru ca cele 4 containere sa aib acces la Internet:
    • Configurai translatarea pe sistemul host astfel nct containerele s poat primi rspunsuri din Internet.
    • Configurai cele 4 containere pentru a putea accesa resurse din Internet pe baza numelor de domeniu al acestora (DNS).
    • Atenie: nu translatai orice pachet este rutat (e.g., cele ntre containere ar trebui s-i vad IP-urile originale)!

Fiierul resolv.conf este gestionat ca volum de ctre Docker :( aceeai poveste ca la exerciiul anterior aceeai soluie (e.g., facei un fiier /etc/resolv.conf.orig pe care l vei suprascrie peste /etc/resolv.conf cu un hook pe up din interfaces).

Ex. 5 [10p] Network Address Translation

  • Configurai reguli de NAT pe sistemele host i Router0 (dup caz, ori unul ori ambele :P), astfel:
    • Conexiunile pe Router0 la porturile (14000 + $E), (14000 + $F) i (14000 + $G) s conduc la conectarea ssh pe sistemele PC1, PC2 respectiv PC3.
    • Conectarea pe host la portul (6000 + $K) s conduc la conectarea pe tracker-ul de pe sistemul PC-X.
  • Sfat: avei grij cum testai: DNAT-ul va funciona DOAR dac venii dintr-o reea extern ruterului ;)

Ex. 6 [10p] Filtrare pachete (iptables)

  • Configurai filtrarea de pachete pe host i / sau Router0 (dup caz) astfel nct:
    • conexiunile SMTP i SSH iniiate de pe sistemul PC3 n afara reelei lui s fie blocate (inclusiv ctre alte rutere!);
    • conexiunile ctre tracker-ul ce ruleaz pe sistemul PC-X s nu fie permise de la PC1, PC2, PC3.
    • blocai TOATE conexiunile externe (i.e., de pe IP-urile din afara staiei) ctre PC2, mai puin protocoalele icmp i ssh.
    • atenie: NU blocai conexiunile iniiate de PC2 i nici rspunsurile de la acestea! folosii reguli stateful (i.e. connection tracking)!

Ex. 7 [10p] Chei SSH

  • Configurai serviciul de SSH pe sistemele PC1, PC2, PC-X i host astfel nct autentificarea cu utilizatorul student de pe oricare sistem s fie permis pe toate celelate sisteme (tot n cadrul utilizatorului student) folosind chei publice;
  • Folosind alias-uri SSH s fie folosite comenzi simplificate pentru conectarea prin SSH ntre sistemele menionate mai sus:
    • ssh pc1 s duc ctre sistemul PC1 cu utilizatorul student;
    • ssh pc2 s duc ctre sistemul PC2 cu utilizatorul student;
    • ssh pc-x s duc ctre sistemul PC-X cu utilizatorul student;
    • ssh host s duc ctre sistemul host cu utilizatorul student;
  • Observaie: ssh-copy-id utilizeaz autentificare ssh pe baz de parol pentru a copia cheia. Maina virtual pe OpenStack nu permite astfel de autentificare pe host, aadar vei fi nevoii s autorizai cheia public manual in fiierul corespunztor!

Atenie: NU V TERGEI CHEIA DE LA OPENSTACK DIN FIIERUL DE AUTORIZAII! V vei pierde accesul la VM.. asigurai-v cnd editai fiierul c nu modificai acea linie!!

Ex. 9 [10p] Descrcare / ncrcare fiiere ntre servere

  • Pe PC3, realizai un script la /home/student/scripts/send-music care va prelua DOAR fiiere care conin DOAR MAJUSCULE cu extensia .mp3 (i.e. [A-Z]+\.mp3, litere mici extensia!) din directorul curent (i.e.: $PWD) i le va uploada pe PC1, n destinaia /home/fs/music; folosii credenialele fs:5$4l0m. NU uitai s-l facei executabil (+ shebang)!
  • Pe host, creai script-ul /root/scripts/copy-firmware care, n primul rnd, descarc DOAR fiierele cu forma [a-zA-Z0-9]+\.bin (caractere alfa-numerice + extensia .bin) de pe staia PC-X din /opt/my-firmware i le ncarc (uploadeaz) pe staia PC2, n directorul /tmp/firmware-files/ (va trebui s-l creai dac nu exist!).
  • Folosi un utilitar aplicabil protocolului (dintre ftp / wget / curl / scp / rsync), dup caz (recomandm studierea tuturor nainte de a lua o decizie!)!

  • Checkerul pentru acest task va refuza s ruleze dac nu exist scripturile / nu sunt executabile (va afia pur i simplu eroarea skipped)!

Ex. 10 [Bonus - 10p] Wireguard tunnelling

  • Dorim s conectm reeaua VLAN10 (cea cu PC1 i PC3, aka un site ntreg) la staia PC-X (endpoint).
    • Vei folosi 2 reele noi: subnetai OPTIM spaiul IPv4 10.66.$J.0/28; asignai prima subreea ctre site-ul VLAN10, iar pe cea de-a doua pentru capetele de tunel (Router0 i PC-X) folosii conveniile de ordinea ale IP-urilor de la primul exerciiu!
    • Denumii interfeele de tunel wg-rl la ambele capete.
    • Pe staiile vizate, toate pachetele destinate ctre 10.66.$J.0/28 trebuie trecute prin acest tunel!
