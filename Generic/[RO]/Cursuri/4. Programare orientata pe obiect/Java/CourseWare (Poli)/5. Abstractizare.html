Laboratorul 5: Abstractizare

Video introductiv: link

Obiective

Scopul acestui laborator este de a nelege principiul abstractizrii, prin prezentarea conceptelor de interfa i de clas abstract i utilizarea lor n limbajul Java.

Conceptele de metode i clase abstracte i de interfee sunt prezente i n alte limbaje OOP, fiecare cu particularitile lor de sintax. Este important ca n urma acestui laborator s nelegei ce reprezint i situaiile n care s le folosii.

Aspectele urmrite sunt:

  • Prezentarea interfeelor i claselor abstracte
  • Motenirea n cazul interfeelor si claselor abstracte
  • Diferenele dintre interfee i clase abstracte

Clasa Instrument nu este instaniat niciodat pentru c scopul su este de a stabili o interfa comun pentru toate clasele derivate. n acelai sens, metodele clasei de baz nu vor fi apelate niciodat. Apelarea lor este ceva greit din punct de vedere conceptual.

Abstractizare

Abstractizarea este unul dintre cele 4 principii POO de baz (Abstractizare, ncapsulare, Motenire, Polimorfism). Abstractizarea nu permite ca anumite caracteristici s fie vizibile n exterior. Acest lucru se refer la construirea unei interfee comune, a unui ablon general pe care se bazeaz o anumit categorie de obiecte, fr a descrie explicit caracteristicile fiecaruia. Obiectele cunosc interfaa comun pe care o au dar nu i cum este ea interpretat de fiecare obiect in parte. Acest lucru este realizat prin utilizarea claselor abstracte i a interfeelor.

Clase abstracte vs Interfee

Folosim o clas abstract atunci cnd vrem:

  • s implementm doar unele dintre metodele din clas
  • ca respectiva clas s nu poat fi instaniat

Folosim o interfa atunci cnd vrem:

  • s avem doar o descriere a structurii, fr implementri
  • ca interfaa n cauz s fie folosit mpreun cu alte interfee n contextul motenirii
Clase abstracte

Dorim s stabilim interfaa comun pentru a putea crea funcionalitate diferit pentru fiecare subtip i pentru a ti ce anume au clasele derivate n comun. O clas cu caracteristicile enumerate mai sus se numete abstract. Crem o clas abstract atunci cnd dorim s:

  • manipulm un set de clase printr-o interfa comun
  • reutilizm o serie metode si membrii din aceast clas in clasele derivate.

Metodele suprascrise n clasele derivate vor fi apelate folosind dynamic binding (late binding). Acesta este un mecanism prin care compilatorul, n momentul n care nu poate determina implementarea unei metode n avans, las la latitudinea JVM-ului (mainii virtuale) alegerea implementrii potrivite, n funcie de tipul real al obiectului. Aceast legare a implementrii de numele metodei la momentul execuiei st la baza polimorfismului. Nu exist instane ale unei clase abstracte, aceasta exprimnd doar un punct de pornire pentru definirea unor instrumente reale. De aceea, crearea unui obiect al unei clase abstracte este o eroare, compilatorul Java semnalnd acest fapt.

Metode abstracte

Pentru a exprima faptul c o metod este abstract (adic se declar doar interfaa ei, nu i implementarea), Java folosete cuvntul cheie abstract: 

abstract void f();

O clas care conine metode abstracte este numit clas abstract. Dac o clas are una sau mai multe metode abstracte, atunci ea trebuie s conin n definiie cuvntul abstract.

Exemplu: 

abstract class Instrument {
...
}

Deoarece o clas abstract este incomplet (exist metode care nu sunt definite), crearea unui obiect de tipul clasei este mpiedicat de compilator.

Clase abstracte n contextul motenirii

O clas care motenete o clas abstract este ea nsi abstract daca nu implementeaz toate metodele abstracte ale clasei de baz. Putem defini clase abstracte care motenesc alte clase abstracte .a.m.d. O clas care poate fi instaniat (adic nu este abstract) i care motenete o clas abstract trebuie s implementeze (defineasc) toate metodele abstracte pe lanul motenirii (ale tuturor claselor abstracte care i sunt prini). Este posibil s declarm o clas abstract fr ca ea s aib metode abstracte. Acest lucru este folositor cnd declarm o clas pentru care nu dorim instane (nu este corect conceptual s avem obiecte de tipul acelei clase, chiar dac definiia ei este complet).

Interfee

Interfeele duc conceptul abstract un pas mai departe. Se poate considera c o interfa este o clas abstract pur: permite programatorului s stabileasc o form pentru o clas (numele metodelor, lista de argumente, valori ntoarse), dar far nicio implementare. O interfa poate conine cmpuri dar acestea sunt n mod implicit static i final. Metodele declarate n interfa sunt n mod implicit public.

Interfaa este folosit pentru a descrie un contract ntre clase: o clas care implementeaz o interfa va implementa metodele definite n interfa. Astfel orice cod care folosete o anumit interfa tie ce metode pot fi apelate pentru aceasta.

Pentru a crea o interfa folosim cuvntul cheie interface n loc de class. La fel ca n cazul claselor, interfaa poate fi declarat public doar dac este definit ntr-un fiier cu acelai nume ca cel pe care l dm acesteia. Dac o interfa nu este declarat public atunci specificatorul ei de acces este package-private. Pentru a defini o clas care este conform cu o interfa anume folosim cuvntul cheie implements. Aceast relaie este asemntoare cu motenirea, cu diferena c nu se motenete comportament, ci doar interfaa. Pentru a defini o interfa care motenete alt interfa folosim cuvntul cheie extends. Dupa ce o interfa a fost implementat, acea implementare devine o clas obinuit care poate fi extins prin motenire.

Iata exemplul dat mai sus, modificat pentru a folosi interfee:

Codul arat astfel: 

interface Instrument {
 
    // Compile-time constant:
    int FIELD = 5; // static & final
 
    // Cannot have method definitions:
    void play(); // Automatically public
 
    String what();
 
    void adjust();
}
 
class WindInstrument implements Instrument {
 
    public void play() {
        System.out.println("WindInstrument.play()");
    }
 
    public String what() {
        return "WindInstrument";
    }
 
    public void adjust() {
    }
}
 
class Trumpet extends WindInstrument {
 
    public void play() {
        System.out.println("Trumpet.play()");
    }
 
    public void adjust() {
        System.out.println("Trumpet.adjust()");
    }
}

Un exemplu pentru o interfa care extinde mai multe interfee: 

interface A{
    void a1();
    void a2();
}
 
interface B {
    int x = 0;
    void b();
}
 
interface C extends A, B {
    // this interface will expose
    //  * all the methods declared in A and B (a1, a2 and b)
    //  * all the fields declared in A and B (x)
}

Implicit, specificatorul de acces pentru membrii unei interfee este public. Atunci cnd implementm o interfa trebuie s specificm c funciile sunt public chiar dac n interfa ele nu au fost specificate explicit astfel. Acest lucru este necesar deoarece specificatorul de acces implicit n clase este package-private, care este mai restrictiv dect public.

Motenire multipl n Java

Interfaa nu este doar o form pur a unei clase abstracte, ci are un scop mult mai nalt. Deoarece o interfa nu specific niciun fel de implementare (nu exist nici un fel de spaiu de stocare pentru o interfa) este normal s combinm mai multe interfee. Acest lucru este folositor atunci cnd dorim s afirmm c X este un A, un B si un C. Acest deziderat este motenirea multipl i, datorit faptului ca o singur entitate (A, B sau C) are implementare, nu apar problemele motenirii multiple din C++. 

interface CanFight {
    void fight();
}
 
interface CanSwim {
    void swim();
}
 
interface CanFly {
    void fly();
}
 
class ActionCharacter {
    public void fight() {
    }
}
 
class Hero extends ActionCharacter implements CanFight, CanSwim, CanFly {
 
    public void swim() {
    }
 
    public void fly() {
    }
}
 
public class Adventure {
 
    static void t(CanFight x) {
        x.fight();
    }
 
    static void u(CanSwim x) {
        x.swim();
    }
 
    static void v(CanFly x) {
        x.fly();
    }
 
    static void w(ActionCharacter x) {
        x.fight();
    }
 
    public static void main(final String[] args) {
        Hero hero = new Hero();
 
        t(hero); // Treat it as a CanFight
 
        u(hero); // Treat it as a CanSwim
 
        v(hero); // Treat it as a CanFly
 
        w(hero); // Treat it as an ActionCharacter
    }
}

Se observ c Hero combin clasa ActionCharacter cu interfeele CanSwim, CanFight, CanFly. Acest lucru se realizeaz specificnd prima data clasa concret (sau abstract) (extends) i abia apoi implements. Metodele clasei Adventure au drept parametri interfeele CanSwin, CanFight, CanFly si clasa ActionCharacter. La fiecare apel de metod din Adventure se face upcast de la obiectul Hero la clasa sau interfaa dorit de metoda respectiv.

Coliziuni de nume la combinarea interfeelor

Combinarea unor interfee care conin o metod cu acelai nume este posibil doar dac metodele nu au tipuri ntoarse diferite i aceeai list de argumente. Totui este preferabil ca n interfee diferite care trebuie combinate s nu existe metode cu acelai nume deoarece acest lucru poate duce la confuzii evidente (sunt amestecate n acest mod 3 concepte: overloading, overriding i implementation).

Extinderea interfeelor

Se pot aduga cu uurin metode noi unei interfee prin extinderea ei ntr-o alt interfa:

Exemplu: 

interface Monster {
    void menace();
}
 
interface DangerousMonster extends Monster {
    void destroy();
}

Deoarece cmpurile unei interfee sunt automat static i final, interfeele sunt un mod convenabil de a crea grupuri de constante, asemntoare cu enum-urile din C , C++.

Iniializarea cmpurilor n interfee

n interfee nu pot exista blank finals (cmpuri final neiniializate) ns pot fi iniializate cu valori neconstante. Cmpurile fiind statice, ele vor fi iniializate prima oar cnd clasa este iniializat.

Exerciii

Contest LambdaChecker: Laborator5

1. (2 puncte) Implementai interfaa Task n cele 3 moduri de mai jos.

  • Un task care s afieze un mesaj la output. Mesajul este specificat n constructor. (OutTask.java)
  • Un task care genereaz un numr aleator de tip int i afieaz un mesaj cu numrul generat la output. Generarea se va realiza n constructor utilizndu-se o instana global a unui obiect de tip Random care a fost iniializat cu seed-ul 12345 Random. (RandomOutTask.java)
  • Un task care incrementeaz un contor global i afieaz valoarea contorului dup fiecare incrementare (CounterOutTask.java).

Not: Acesta este un exemplu simplu pentru Command Pattern

2. (3 puncte) Interfaa Container specific interfaa comun pentru colecii de obiecte Task, n care se pot aduga i din care se pot elimina elemente. Creai dou tipuri de containere care implementeaz aceast clas:

  1. Stack - care implementeaz o strategie de tip LIFO
  2. Queue - care implementeaz o strategie de tip FIFO

Bonus: Incercai s evitai codul similar n locuri diferite!

Hint: Putei reine intern colecia de obiecte, utiliznd clasa ArrayList din SDK-ul Java. 

ArrayList<Task> list = new ArrayList<Task>();

3. (2 puncte) Creai 4 interfee: Minus, Plus, Mult, Div care conin cte o metod aferent numelui ce are ca argument un numar de tipul float.

Spre exemplu interfata Minus va declara metoda: 

void minus(float value);

Scriei clasa Operation care s implementeze toate aceste interfee. Pe lnga metodele interfeelor implementate aceasta va conine:

  • un cmp number de tipul float asupra cruia se vor efectua operaiile implementate
  • o metodata getter getNumber care va returna valoarea cmpului value
  • un constructor care primete ca parametru valoarea iniial pentru campul value

Pentru cazul n care metoda div este apelat cu valoare 0 operaia nu va fi efectuat i se va afia mesajul Division by 0 is not possible.

4. (3 puncte) Implementai clasa Song i clasa abstracta Album.

Song:

  • stocheaz atributele name: String, id: int, composer: String
  • implementeaz un constructor care va iniializa atributele specificate anterior
  • implementeaz metodele de tip getter i setter pentru fiecare atribut
  • suprascrie metoda toString() care va returna un String forma Song{name=name, id=id, composer=composer}

Album:

  • stocheaz o list de cntece (putei folosi ArrayList)
  • declar metoda abstract void addSong(Song song)
  • implementeaz metodele void removeSong(Song song)
  • suprascrie metoda toString() care va returna un String forma Album{songs=[Song, Song, ]}

Dup implementarea claselor Song i Album creai clasele DangerousAlbum, ThrillerAlbum i BadAlbum care vor moteni clasa Album i vor implementa metoda addSong dup urmtoarele reguli:

  • DangerousAlbum conine doar melodii cu id-ul numr prim
  • ThrillerAlbum conine melodii scrise doar de Michael Jackson i au id-ul numr par
  • BadAlbum conine doar melodii care au nume de 3 litere i id numr palindrom

n cazul n care criteriul de adaugare specific unui album nu este respectat melodia nu va fi adaugat n acesta.
