Introducere

Design pattern-urile reprezint soluii generale i reutilizabile ale unei probleme comune n design-ul software. Un design pattern este o descriere a soluiei sau un template ce poate fi aplicat pentru rezolvarea problemei, nu o bucata de cod ce poate fi aplicata direct. n general pattern-urile orientate pe obiect arat relaiile i interaciunile dintre clase sau obiecte, fr a specifica ns forma final a claselor sau a obiectelor implicate.

Se consider c exist aproximativ 2000 de design patterns [2], iar principalul mod de a le clasifica este urmtorul:

  • Gang of Four patterns:
  • Concurrency patterns
  • Architectural patterns - sunt folosite la un nivel mai inalt decat design patterns, stabilesc nivele i componente ale sistemelor/aplicaiilor, interaciuni ntre acestea (e.g. Model View Controller i derivatele sale). Acestea descriu structura ntregului sistem, iar multe framework-uri vin cu ele deja ncoporate, sau faciliteaza aplicarea lor (e.g. Java Spring). n cadrul laboratoarelor nu ne vom lega de acestea.

O carte de referin pentru design patterns este Design Patterns: Elements of Reusable Object-Oriented Software [1], denumit i Gang of Four (GoF). Aceasta definete 23 de design patterns, foarte cunoscute i utilizate n prezent. Aplicaiile pot ncorpora mai multe pattern-uri pentru a reprezenta legturi dintre diverse componente (clase, module). n afar de GoF, i ali autori au adus n discuie pattern-uri orientate n special pentru aplicaiile enterprise i cele distribuite.

Pattern-urile GoF sunt clasificate n felul urmtor:

  • Creational Patterns - definesc mecanisme de creare a obiectelor
    • Singleton, Factory etc.
  • Structural Patterns - definesc relaii ntre entiti
    • Decorator, Adapter, Facade, Composite, Proxy etc.
  • Behavioural Patterns - definesc comunicarea ntre entiti
    • Visitor, Observer, Command, Mediator, Strategy etc.

Design pattern-urile nu trebuie privite drept nite reete care pot fi aplicate direct pentru a rezolva o problem din design-ul aplicaiei, pentru c de multe ori pot complica inutil arhitectura. Trebuie nti neles dac este cazul s fie aplicat un anumit pattern, si de-abia apoi adaptat pentru situaia respectiv. Este foarte probabil chiar s folosii un pattern (sau o abordare foarte similar acestuia) fr s v dai seama sau s l numii explicit. Ce e important de reinut dup studierea acestor pattern-uri este un mod de a aborda o problem de design.

n laboratoarele precedente au fost descrise patternurile Singleton i Visitor. Singleton este un pattern creaional, simplu, a crui folosire este controversat (vedei n laborator explicaia cu anti-pattern). Visitor este un pattern comportamental, i dup cum ai observat ofer avantaje n anumite situaii, n timp ce pentru altele nu este potrivit. Pattern-urile comportamentale modeleaz interaciunile dintre clasele i componentele unei aplicaii, fiind folosite n cazurile n care vrem sa facem un design mai clar i uor de adaptat i extins.

Factory

Patternurile de tip Factory sunt folosite pentru obiecte care genereaz instane de clase nrudite (implementeaz aceeai interfa, motenesc aceeai clas abstract). Acestea sunt utilizate atunci cnd dorim s izolm obiectul care are nevoie de o instan de un anumit tip, de creearea efectiv acesteia. n plus clasa care va folosi instana nici nu are nevoie s specifice exact subclasa obiectului ce urmeaz a fi creat, deci nu trebuie s cunoasc toate implementrile acelui tip, ci doar ce caracteristici trebuie s aib obiectul creat. Din acest motiv, Factory face parte din categoria Creational Patterns, deoarece ofer o soluie legat de creearea obiectelor.

Aplicabilitate:

  • n biblioteci/API-uri, utilizatorul este separat de implementarea efectiv a tipului i trebuie sa foloseasc metode factory pentru a obine anumite obiecte. Clase care ofer o astfel de funcionalitate putei gsi i in core api-ul de Java, in api-ul java.nio (e.g. clasa FileSystems), n Android SDK (e.g. clasa SocketFactory) etc.
  • Atunci cnd crearea obiectelor este mai complex (trebuie realizate mai multe etape etc.), este mai util s separm logica necesar instanierii subtipului de clasa care are nevoie de acea instan. Asta nseamn c putei folosi metode factory care s v construiasc obiectul i dac avei doar un tip, nu mai multe.

Factory Method Pattern

Folosind pattern-ul Factory Method se poate defini o interfa pentru crearea unui obiect. Clientul care apeleaz metoda factory nu tie/nu l intereseaz de ce subtip va fi la runtime instana primit.

Spre deosebire de Abstract Factory, Factory Method ascunde construcia unui obiect, nu a unei familii de obiecte inrudite, care extind un anumit tip. Clasele care implementeaz Abstract Factory conin de obicei mai multe metode factory.

Situaia cea mai ntlnit n care se potrivete acest pattern este aceea cnd trebuie instaniate multe clase care implementeaz o anumit interfa sau extind o alt clas (eventual abstract), ca n exemplul de mai jos. Singleton Factory

De obicei avem nevoie ca o clas factory s fie utilizat din mai multe componente ale aplicaiei. Ca s economisim memorie este suficient s avem o singur instan a factory-ului i s o folosim pe aceasta. Folosind pattern-ul Singleton putem face clasa factory un singleton, i astfel din mai multe clase putem obine instan acesteia.

Un exemplu ar fi Java Abstract Window Toolkit (AWT) ce ofer clasa abstract java.awt.Toolkit care face legtura dintre componentele AWT i implementrile native din toolkit. Clasa Toolkit are o metod factory Toolkit.getDefaultToolkit() ce ntoarce subclasa de Toolkit specific platformei. Obiectul Toolkit este un Singleton deoarece AWT are nevoie de un singur obiect pentru a efectua legturile i deoarece un astfel de obiect este destul de costisitor de creat. Metodele trebuie implementate n interiorul obiectului i nu pot fi declarate statice deoarece implementarea specific nu este cunoscut de componentele independente de platform.

Observer Pattern

Design Pattern-ul Observer definete o relaie de dependen 1 la n ntre obiecte astfel nct cnd un obiect i schimb starea, toi dependenii lui sunt notificai i actualizai automat. Folosirea acestui pattern implic existena unui obiect cu rolul de subiect, care are asociat o list de obiecte dependente, cu rolul de observatori, pe care le apeleaz automat de fiecare dat cnd se ntmpl o aciune.

Acest pattern este de tip Behavioral (comportamental), deorece faciliteaz o organizare mai bun a comunicaiei dintre clase n funcie de rolurile/comportamentul acestora.

Aplicabilitate

Observer se folosete n cazul n care mai multe clase(observatori) depind de comportamentul unei alte clase(subiect), n situaii de tipul:

  • o clas implementeaz/reprezint logica, componenta de baz, iar alte clase doar folosesc rezultate ale acesteia (monitorizare).
  • o clas efectueaz aciuni care apoi pot fi reprezentate n mai multe feluri de ctre alte clase (view-uri ca n figur de mai jos).

Practic n toate aceste situaii clasele Observer observ modificrile/aciunile clasei Subject. Observarea se implementeaz prin notificri iniiate din metodele clasei Subject.

Structur

Pentru aplicarea acestui pattern, clasele aplicaiei trebuie s fie structurate dup anumite roluri, i n funcie de acestea se stabilete comunicarea dintre ele. n exemplul din figure 3 , avem dou tipuri de componente, Subiect i Observator, iar Observator poate fi o interfa sau o clas abstract ce este extins cu diverse implementri, pentru fiecare tip de monitorizare asupra obiectelor Subiect.

  • observatorii folosesc datele subiectului
  • observatorii sunt notificai automat de schimbri ale subiectului
  • subiectul cunoate toi observatorii
  • subiectul poate aduga noi observatori

Subiect

  • nu trebuie s tie ce fac observatorii, trebuie doar s menin referine ctre obiecte de acest tip
  • nu tie ce fac observatorii cu datele
  • ofer o metod de adugare a unui Observator, eventual i o metod prin care se pot deinregistra observatori
  • menine o list de referine cu observatori
  • cnd apar modificri (e.g. se schimb starea sa, valorile unor variabile etc) notific toi observatorii

Observator

  • definete o interfa notificare despre schimbri n subiect
  • ca implementare:
    • toi observatorii pentru un anumit subiect trebuie s implementeze aceast interfa
    • ofer una sau mai multe metode care s poat fi invocate de ctre Subiect pentru a notifica o schimbare. Ca argumente se poate primi chiar instana subiectului sau obiecte speciale care reprezint evenimentul ce a provocat schimbarea.

Strategy Pattern

Design pattern-ul Strategy ncapsuleaz algoritmii n clase ce ofer o anumit interfa de folosire, i pot fi selecionai la runtime. Ca i Command, acest pattern este behavioral pentru ca permite decuplarea unor clase ce ofer un anumit comportament i folosirea lor independent n funcie de situaia de la runtime.

Acest pattern este recomandat n cazul n care avem nevoie de un tip de algoritm (strategie) cu mai multe implementri posibile si dorim s alegem dinamic care algoritm l folosim, fr a face sistemul prea strns cuplat.

Exemple de utilizare:

  • sisteme de tip Layout Managers din API-urile pentru UI
  • selectarea in mod dinamic la runtime a unor algoritmi de sortare, compresie, criptare etc.

Structur:

  • trebuie s definii o interfa comun pentru strategiile pe care le implementai (fie ca o interface sau ca o clasa abstract)
  • implementai strategiile respectnd interfaa comun
  • clasa care are nevoie s foloseasc strategiile va ti doar despre interfaa lor, nu va fi legat de implementrile concrete

Denumirile uzuale n exemplele acestui pattern sunt: Strategy (pt interfa sau clasa abstract), ConcreteStrategy pentru implementare, Context, clasa care folosete/execut strategiile.

Recomandare: Urmrii link-ul de la referine ctre postul de pe Stack Overflow care descrie necesitatea pattern-ului Strategy. Pe lng motivul evident de ncapsulare a prelucrrilor/algoritmilor (care reprezint strategiile efective), se prefer o anumit abordare: la runtime se verific mai multe condiii i se decide asupra strategiei. Concret, folosind mecanismul de polimorfism dinamic, se folosete o anumit instan a tipului de strategie (ex. Strategy str = new CustomStrategy), care se paseaz n toate locurile unde este nevoie de Strategy. Practic, n acest fel, utilizatorii unei anumite strategii vor deveni agnostici n raport cu strategia utilizat, ea fiind instaniat ntr-un loc anterior i putnd fi gata utilizat. Gndii-v la browserele care trebuie s detecteze dac device-ul este PC, smartphone, tablet sau altceva i n funcie de acest lucru s randeze n mod diferit. Fiecare randare poate fi implementat ca o strategie, iar instanierea strategiei se va face ntr-un punct, fiind mai apoi pasat n toate locurile unde ar trebui s se in cont de aceast strategie.

Command

Design pattern-ul Command incapsuleaz un apel cu tot cu parametri ntr-o clas cu interfat generic. Acesta este Behavioral pentru c modific interaciunea dintre componente, mai exact felul n care se efectueaz apelurile.

Acest pattern este recomandat n urmtoarele cazuri:

  • pentru a uura crearea de structuri de delegare, de callback, de apelare intarziat
  • pentru a reine lista de comenzi efectuate asupra obiectelor
  • accounting
  • liste de Undo, Rollback pentru tranzacii-suport pentru operaii reversibile (undoable operations)

Exemple de utilizare:

  • sisteme de logging, accounting pentru tranzacii
  • sisteme de undo (ex. editare imagini)
  • mecanism ordonat pentru delegare, apel ntrziat, callback

Funcionare i necesitate

n esent, Command pattern (aa cum v-ai obinuit i lucrnd cu celelate Pattern-uri pe larg cunoscute) presupune ncapsularea unei informaii referitoare la aciuni/comenzi folosind un wrapper pentru a ine minte aceast informaie i pentru a o folosi ulterior. Astfel, un astfel de wrapper va deine informaii referitoare la tipul aciunii respective (n general un asemenea wrapper va expunde o metod execute(), care va descrie comportamentul pentru aciunea respectiv).

Mai mult nc, cnd vorbim de Command Pattern, n terminologia OOP o s ntlnii deseori i noiunea de Invoker. Invoker-ul este un middleware ca funcionalitate care realizeaz managementul comenzilor. Practic, un Client, care vrea s fac anumite aciune, va instania clase care implementeaz o interfa Command. Ar fi incomod ca, n cazul n care aceste instanieri de comenzi provin din mai multe locuri, acest management de comenzi s se fac local, n fiecare parte (din raiuni de economie, nu vrem s duplicm cod). Invoker-ul apare ca o necesitate de a centraliza acest proces i de a realiza intern management-ul comenzilor (le ine ntr-o list, ine cont de eventuale dependene ntre ele, totul n funcie de context).

Un client (generic spus, un loc de unde se lanseaz comenzi) instaniaz comenzile i le paseaz Invoker-ului. Din acest motiv Invoker-ul este un middleware ntre client i receiver, fiindc acesta va apela execute pe fiecare Command, n funcie de logica s intern.

Structura

Ideea principal este de a crea un obiect de tip Command care va reine parametrii pentru comand. Comandantul reine o referin la comand i nu la componenta comandat. Comanda propriu-zis este anunat obiectului Command (de ctre comandant) prin execuia unei metode specificate asupra lui. Obiectul Command este apoi responsabil de trimiterea (dispatch) comenzii ctre obiectele care o ndeplinesc (comandai).

Tipuri de componente (roluri):

  • Invoker - comandantul
    • apeleaz aciuni pe comenzi (invoc metode oferite de obiectele de tip Command)
    • poate menine, dac e cazul, o list a tutoror comenzilor aplicate pe obiectul (obiectele) comandate. Este necesar reinerea acestei liste de comenzi atunci cnd implementm un comportament de undo/redo al comenzilor.
    • primete clase Command pe care s le invoce
  • Receiver - comandatul
    • este clasa asupra creia se face apelul
    • conine implementarea efectiv a ceea ce se dorete executat
  • Command - obiectele pentru reprezentarea comenzilor implementeaz aceast interfa/o extind dac este clas abstract
    • concrete command - ne referim la implementri/subclasele acesteia
    • de obicei conin metode cu nume sugestiv pentru executarea aciunii comenzii (e.g. execute()). Implementrile acestora conin apelul ctre clasa Receiver.
    • n cazul implementrii unor aciuni undoable adugm metode pentru undo i/sau redo.
    • in referine ctre comandai (receivers) pentru a aplica/invoca aciunea ce reprezint acea comand

n Java, se pot folosi att interfee ct i clase abstracte, pentru Command, depinznd de situaie (e.g. clas abstract dac tim sigur ca obiectele de tip Command nu mai au nevoie s extind i alte clase).

Command va fi o interfa, creia pe lang metoda de execute() i vom asocia i o metod de undo(). 

interface Command {
   void execute();
 
   void undo();
}

BlurCommand i ResizeCommand vor implementa interfaa Command. La apelul execute(), BlurCommand va modifica cmpul blurStrength din clasa Image, iar ResizeCommand va modifica dimensiunea, lungimea i nlimea imaginii. ntruct ne dorim s implementm un mecanism de undo, este nevoie s reinem valoare anterioar. 

// Concrete command
 
public class BlurCommand implements Command {
   private final Image image;
   private int previousBlurStrength;
   private int nextBlurStrength;
 
   public BlurCommand(Image image, int blurStrength) {
       this.image = image;
       this.nextBlurStrength = blurStrength;
   }
 
   @Override
   public void execute() {
       previousBlurStrength = image.getBlurStrength();
       image.setBlurStrength(nextBlurStrength);
   }
 
   @Override
   public void undo() {
       nextBlurStrength = previousBlurStrength;
       previousBlurStrength = image.getBlurStrength();
       image.setBlurStrength(nextBlurStrength);
   }
}
 
public class ResizeCommand implements Command {
   private final Image image;
   private int previousWidth;
   private int previousLength;
   private int nextWidth;
   private int nextLength;
 
   public ResizeCommand(Image image, int width, int length) {
       this.image = image;
       nextWidth = width;
       nextLength = length;
   }
 
   @Override
   public void execute() {
       previousWidth = image.getWidth();
       image.setWidth(nextWidth);
 
       previousLength = image.getLength();
       image.setLength(nextLength);
   }
 
   @Override
   public void undo() {
       nextWidth = previousWidth;
       previousWidth = image.getWidth();
       image.setWidth(nextWidth);
 
       nextLength = previousLength;
       previousLength = image.getLength();
       image.setLength(nextLength);
   }
}

Invoker-ul este clasa Editor. Aceasta va avea dou metode, edit si undo, care vor fi apelate de ctre user. n plus, vom pstra n aceast clas i o list a comenzilor aplicate pe imagine. Cu acest list vom putea implementm un comportament de undo.
Metoda edit va primi ca parametru o referi la o instan command, apoi va fi iniiat o cerere de ctre Editor prin apelarea metodei execute(), cernd astfel execuia comenzii. 

// Invoker
 
public class Editor {
   // LinkedList este folosit ca stiv n Java
   private LinkedList<Command> history = new LinkedList<>(); // pstreaz comenzile aplicate pe imagine
 
   public void edit(Command command) {
       history.push(command);
       command.execute();
   }
 
   public void undo() {
       if (history.isEmpty()) return;
 
       Command command = history.pop();
       if (command != null) {
           command.undo();
       }
   }
}

    public Pizza(String pizzaSize, String crust, int cheeseCount, int pepperoniCount) {
        this(pizzaSize, crust, cheeseCount, pepperoniCount, 0, 0);
    }
 
    public Pizza(String pizzaSize, String crust, int cheeseCount, int pepperoniCount, int hamCount) {
        this(pizzaSize, crust, cheeseCount, pepperoniCount, hamCount, 0);
    }
 
    public Pizza(String pizzaSize, String crust, int cheeseCount, int pepperoniCount, int hamCount, int mushroomsCount) {
        this.pizzaSize = pizzaSize;
        this.crust = crust;
        this.cheeseCount = cheeseCount;
        this.pepperoniCount = pepperoniCount;
        this.hamCount = hamCount;
        this.mushroomsCount = mushroomsCount;
    }
 
    // getters
}

Gndii-v ce se va ntampla dac se schimb ordinea parametrilor. Acest lucru minor va strica funcionalitatea complet a creative unei instane de Pizza.

n concluzie, modelul de constructori supraincarcati funcioneaz, dar este greu de meninut dac se schimb funcionalitatea i introducem noi parametri, numrul constructorilor va crete, de asemenea.

Folosirea de getters i setters
Pizza.java
Pizza pizza = new Pizza();
 
pizza.setPizzaSize("small");
pizza.setCrust("thin");
pizza.setMushroomsCount(1);
pizza.setCheeseCount(1);
 
// do something with pizza

Aceast soluie nu prezint niciunul dintre dezavantajele modelului anterior. Este uor s scalai clasa, mai uor de instaniat, mai uor de citit i mai flexibil. Modelul are ns dezavantaje grave. Construcia clasei este mprit n apeluri multiple, prin urmare instana poate fi ntr-o stare parial construit / invalid.

Folosirea builder pattern
Pizza.java
public class Pizza {
    private String pizzaSize;
    private String crust;
    private int cheeseCount;
    private int pepperoniCount;
    private int hamCount;
    private int mushroomsCount;
 
    public static class Builder {
        private String pizzaSize; // mandatory
        private String crust; // mandatory
        private int cheeseCount = 0; // optional
        private int pepperoniCount = 0; // optional
        private int hamCount = 0; // optional
        private int mushroomsCount = 0; // optional
 
        public Builder(String pizzaSize, String crust) {
            this.pizzaSize = pizzaSize;
            this.crust = crust;
        }
 
        public Builder cheeseCount(int cheeseCount) {
            this.cheeseCount = cheeseCount;
            return this;
        }
 
        public Builder pepperoniCount(int pepperoniCount) {
            this.pepperoniCount = pepperoniCount;
            return this;
        }
 
        public Builder hamCount(int hamCount) {
            this.hamCount = hamCount;
            return this;
        }
 
        public Builder mushroomsCount(int mushroomsCount) {
            this.mushroomsCount = mushroomsCount;
            return this;
        }
 
        public Pizza build() {
            return new Pizza(this);
        }
    }
 
    private Pizza(Builder builder) {
        this.pizzaSize = builder.pizzaSize;
        this.crust = builder.crust;
        this.cheeseCount = builder.cheeseCount;
        this.pepperoniCount = builder.pepperoniCount;
        this.hamCount = builder.hamCount;
        this.mushroomsCount = builder.mushroomsCount;
    }
 
    // getters
}

Am fcut constructorul privat, astfel nct clasa noastr s nu poat fi instaniat direct. n acelai timp am adugat o clas static Builder cu un constructor care are parametrii notri obligatori pizzaSize i crust, metode de setare a parametrilor opionali i, n final, o metod build() metoda care va returna o nou instan a clasei Pizza. Metodele setter returneaz instana de builder n sine, oferindu-ne astfel o interfa fluent cu metoda de nlnuire.

Pizza.java
Pizza pizza = new Pizza.Builder("large", "thin")
    .cheeseCount(1)
    .pepperoniCount(1)
    .build();

Este mult mai uor s scriei i, mai important, s citii acest cod. La fel ca n cazul constructorului, putem verifica parametrii trecui pentru orice nclcare, cel mai adesea n cadrul metodei build() sau a metodei setter, i putem arunca IllegalStateException dac exist nclcri nainte de a crea o instan a clasei.

Modelul Builder are unele dezavantaje minore. n primul rnd, trebuie s creai un obiect Builder nainte de a crea obiectul clasei n sine. Aceasta ar putea fi o problem n unele cazuri critice de performan iar clasa devine puin mai mare cnd vine vorba de liniile de cod.

n ansamblu, modelul Builder este o tehnic foarte frecvent utilizat pentru crearea obiectelor i este o alegere bun de utilizat atunci cnd clasele au constructori cu parametri multipli (n special opionali) i este posibil s se schimbe n viitor. Codul devine mult mai uor de scris i de citit n comparaie cu constructorii supraincarcati, iar clasa este la fel de bun ca folosirea de getters i setters, dar este mult mai sigur.

Summary

Principii de design adresate de aceste patternuri:

  • Dependency Injection Principle - componentele trebuie s depind de tipuri abstracte, nu de implementri
    • Factory respect acest principiu, componentele depinznd de interfaa pentru un tip, nu de un subtip anume
  • Separarea codului care se schimb de cel care rmne la fel - n cazul Strategy folosim o interfa pentru strategii, depindem de aceea, si putem schimba implementrile fr a modifica codul care le folosete (exemplu)
  • Loosly coupled design - n cazul Observer componentele sunt slab legate ntre ele

Task 1 - Observer, Strategy, Factory (8p)

Part 1. (4p) - Strategy, Factory

n cadrul acestui exerciiu, dorim s implementm un magazin, ce are disponibile mai multe modaliti de plat pentru clienii si (folosind un voucher, IBAN-ul sau un card pe care l posed).

Cele 3 modaliti de plat sunt reprezentate de clase ce implementeaz interfaa PaymentStrategy.

Trebuie s implementai:

  • Metoda pay din cadrul fiecrei metode de plat.
  • Metoda getPaymentMethod, ce primete un client ca parametru i instaniaz o tranzacie folosind o anumit metod de plat, aleas random.

HINT: getPaymentMethod va fi folosit n cadrul unei metode ce trebuie implementat n cadrul prii de Observer

Part 2. (4p) - Observer

n acest context, subiectul (cel care este Observable) este clasa Person. Aceast clas implementeaz interfaa BalanceObserver (care este Observer-ul). Clasa Shop (ce joac rol de Publisher, ntruct implementeaz interfaa TransactionPublisher) va avea rolul de a notifica un client cnd o tranzacie este efectuat.

Trebuie s implementai:

  • Motoda update din cadrul clasei Person, ce notific faptul c un client are n cont mai puini Lei dect o anumit limit, specific fiecrui client n parte, printr-un mesaj adecvat (<nume> <prenume>, ramai fara bani, saracule!).
  • Metoda payBy din cadrul clasei Person, ce ntoarce valoarea de adevr a reuitei efecturii unei pli (dac e posibil efectueaz plata), iar n caz negativ afiseaz i un mesaj adecvat (<nume> <prenume>, pleaca de aici, saracule!).
  • Metoda createTransaction din cadrul clasei Shop, ce verific dac trebuie notificat persoana cu ajutorul metodei update, n urma unei tranzacii.

HINT: trebuie folosit i metoda payBy

Task 2 - Builder pattern (2p)

a) Scriei cmpuri n skeletul clasei House pentru cteva faciliti obligatorii n construcia unei case, spre exemplu locaia construciei, numrul de etaje, nclzire, camere dar i unele opionale pe care le poate selecta sau nu clientul, cum ar fi electrocasnice, piscin, panouri solare, securitate etc.

Completai constructorul privat, metodele de get i metoda toString.

b) n clasa de builder, completai cmpurile, constructorul i metodele de adugare a facilitilor opionale.

c) Finalizai metoda build i testai funcionalitatea ntr-o clas Main creat de voi, acoperind cazuri n care se construiete o casa doar cu facilitai obligatorii i altele adugnd i pe cele opionale.

Task 3 - Command pattern (8p)

Implementai folosind patternul Command un editor de diagrame foarte simplificat. Scheletul de cod conine o parte din clase i cteva teste.

Componentele principale ale programului:

  • DiagramCanvas - reprezint o diagram care conine obiecte de tip DiagramComponent
  • DrawCommand - interfa pentru comenzile fcute asupra diagramei sau a componentelor acesteia
  • Invoker - primete comenzile i le execut
  • Client - entry-point-ul n program

(4p) Implementai 5 tipuri de comenzi, pentru urmtoarele aciuni:

  • Draw rectangle - creaz o DiagramComponent i o adaug n DiagramCanvas
  • Resize - modific width i height al unei DiagramComponent pe baza unui procent dat
  • Change color - modific culoarea unei DiagramComponent
  • Change text - modific textul unei DiagramComponent
  • Connect components - conecteaz o DiagramComponent la alta

Implementai pe Invoker metoda execute() care va executa comanda primit ca argument.

Comenzile primesc n constructor referina ctre DiagramCanvas i alte argumente necesare lor. De exemplu, comanda pentru schimbarea culorii trebuie sa primeasc i culoarea nou i indexul componentei.

Pentru acest task nu este nevoie s implementai i metoda undo(), doar execute().

Comenzile implementeaz n afar de metodele interfeei i metoda toString() pentru a afia comanda. Recomandm folosirea IDE-ului pentru a o genera.

(6p) Implementai n comenzi i n Invoker mecanismul de undo/redo al comenzilor. Recomandm n Invoker sa folosii dou structuri de date, una care s menin comenzile efectuate, iar una pentru comenzile fcute undo. Metoda reset() de pe Invoker va avea ca scop resetarea tuturor membrilor acestuia.
